# ТЗ — График Ганта v2 «Mission Control»

> Новая вкладка `/gantt-v2` рядом со старой `/gantt` (старая не трогается, остаётся как есть). Целевой пользователь — Алёна (собственник, оборот 2 млрд ₽), команда — 11 человек, 100+ активных заказов в работе одновременно. Заходим с чистого листа, переиспользуя из старой вкладки только утилиты (`colorHexFromName`, типы данных).

---

## 0. Промпт, по которому написано это ТЗ

> Представь себя главой продуктового отдела бренда с оборотом 2 млрд ₽ и ассортиментом ~150 SKU/сезон. У тебя на руках одновременно: 80–120 активных заказов на производство, 30–50 заказов упаковки, 20–30 фасонов в разработке. Ты заходишь утром в систему и за 30 секунд должна понять:
>
> 1. Что горит **прямо сейчас** — где сегодня надо вмешаться.
> 2. Какие заказы выпадают из плана запуска.
> 3. Где узкие места: фабрика, логист, PM перегружены.
> 4. Какие заказы скоро финиширует (готовь упаковку/контент).
> 5. Что в разработке близко к запуску.
>
> Спроектируй один экран — график Ганта по всему циклу товара (Разработка → Производство → ОТК → Доставка). Не «график-визуализация», а рабочий инструмент управления: фильтры, группировки, мульти-выбор, быстрые действия по hover, цветовое кодирование рисков, KPI-полоса. Думай как Linear / Notion Timeline / GitHub Projects, не как MS Project.
>
> Ключевые ограничения:
> - Алёна **не программист** — простой язык в UI, без жаргона.
> - На экране может быть 100+ строк → виртуализация, плотный режим, фильтры по умолчанию должны скрывать «спокойные» заказы.
> - Команда смотрит на один экран, ролевой фильтрации нет (см. memory `feedback_shared_dashboard`).
> - Drag за края (◀ ▶) уже введён как закон — не ломаем (см. memory `feedback_gantt_drag_edges`).
> - Даты — ползунком, не календарём (memory `feedback_drag_dates`).
> - Цвета — с hex-кружком (memory `feedback_color_chip`).
>
> Опиши: пользовательские сценарии, информационную архитектуру экрана, фильтры/группировки, таймлайн-визуал, поведение строк, KPI-полосу, mobile-версию. Приоритизируй P0 / P1 / P2.

## 1. Цель и метрика успеха

**Цель.** Один экран, на котором руководитель за < 30 сек понимает состояние всего цикла товара и может действовать (открыть карточку, изменить дату, переназначить ответственного) без перехода в другие разделы.

**Что считаем успехом:**
- Алёна заходит утром → видит «горящие 8» вверху → за 5 минут принимает решения по ним.
- Не нужно переходить в `/orders`, `/packaging-orders`, `/models` для рутинной проверки.
- Можно ответить на вопрос «что мы запускаем в июне?» одним фильтром.
- Команда (Вера, Оля, Настя) использует тот же экран — не нужны отдельные «мои задачи».

## 4. Функциональные требования

### 4.1. Шапка экрана (P0)

- **Заголовок** + счётчик строк после фильтров.
- **Кнопка «+ Новый заказ»** (открывает форму создания, преселектит фильтр-контекст: если выбран бренд/PM — подставляет).
- **Меню «⋯»**: Экспорт PDF/PNG (срез текущего вида), Экспорт Excel (плоская таблица заказов), Печать.

### 4.2. KPI-полоса (P0)

Шесть «плиток», sticky. Каждая — кликабельный фильтр.

| Плитка | Считается | Клик → |
|---|---|---|
| **В работе** | Заказы в статусах кроме ON_SALE/CANCELLED | Сбросить фильтры |
| **Горит сейчас** | Заказы, у которых текущая фаза просрочена ИЛИ дедлайн через ≤ 5 дн без отметки done | Применить фильтр «Горит» |
| **Просрочено** | Заказы, где end < today и done=false | Фильтр «Просрочено» |
| **На этой неделе** | Заказы, у которых в эту неделю происходит финиш фазы | Фильтр «эта неделя» |
| **Цикл-тайм средний** | Средний фактический цикл по закрытым за 60 дн (decisionDate → arrivalPlannedDate) | — |
| **Фабрики** | Сколько фабрик в перегрузе (>X одновременных заказов) | Фильтр на эти фабрики |

«Горит» считается по следующему правилу: фаза активная (предыдущие done) И (end<today ИЛИ end<=today+5дн).

### 4.3. Пресеты (P1)

Сохранённые комбинации (фильтр + группировка + сортировка + зум). По умолчанию преднастроенные:
- ⭐ **Горит сейчас** — фильтр «Горит» + сортировка по дате просрочки.
- **Все мои** — owner = current user.
- **Июнь-запуск** — launchMonth = текущий+2.
- **Гуанчжоу** — фабрика страна = CN.

Алёна может сохранить свой пресет (имя + иконка). Хранятся в БД на пользователе (поле `userPreferences` JSON).

### 4.4. Фильтры (P0)

Все — с мульти-выбором, цветовым кодом, поиском внутри:

| Фильтр | Источник |
|---|---|
| Бренд | enum: White One / Сердцебиение |
| Этап | Разработка / Производство / ОТК / Доставка / Упаковка / Разработка фасона |
| Ответственный (PM) | Активные пользователи с ролью OWNER/PM/ASSISTANT/CONTENT/LOGIST |
| Фабрика | Из справочника, с флагом страны |
| Месяц запуска | yyyy-mm chips с количеством заказов |
| Статус | 10 статусов заказа + статусы упаковки |
| Категория | Из фасона |
| Поиск | По номеру заказа, фасону, цвету, артикулу |

**Поведение:**
- Фильтры комбинируются по AND.
- Активные фильтры показаны чипами под строкой фильтров, можно убрать × .
- «Сбросить всё» появляется когда есть ≥1 активный.
- URL-state: фильтры в querystring (shareable link).

### 4.5. Группировка (P0)

Радио-выбор, одна группировка одновременно:
- **Без группировки** — плоский список, сортировка по дедлайну.
- **По бренду** (White One / Сердцебиение).
- **По фабрике** — отдельные группы + «Без фабрики».
- **По ответственному (PM)**.
- **По месяцу запуска** — `2026-06`, `2026-07`, `Без даты`.
- **По этапу** — все заказы в Производстве вместе и т.д.
- **По категории фасона**.

Группа сворачивается/разворачивается. У каждой:
- Имя группы + бейдж количества.
- **Мини-загрузка**: горизонтальный bar — сколько в каком статусе (визуально как pie, но как stacked bar).
- Сворачивается по клику на стрелку, состояние сохраняется в localStorage.

### 4.6. Сортировка (P0)

Внутри группы (или плоского списка):
- **По дедлайну** (по умолчанию: ближайший финиш цикла наверху).
- **По дате запуска** (launchMonth).
- **По степени срочности** — горящие сверху, потом просроченные, потом по дедлайну.
- **По названию фасона**.

### 4.7. Тип вида (P1)

Переключатель сверху-справа:
- **Гант** (по умолчанию) — то, что описано выше.
- **Список** — таблица с колонками: фото, название, фасон, цвет, кол-во, PM, фабрика, статус, дедлайн, дни остатка. Сортируемые колонки. Полезен на маленьких экранах и при экспорте.
- **Канбан** (P2) — колонки = этапы, карточки = заказы. Drag между колонками меняет статус. Для команды Веры/Оли удобнее.

### 4.8. Зум и плотность (P0)

- **Зум** на временной шкале: 1 нед / 1 мес / 3 мес / 6 мес / 1 год / Авто. Слайдер или кнопки, состояние в URL.
- **Плотность строк**: компакт (h-5, без фото-превью) / стандарт (h-9, фото 32px) / просторный (h-14, фото 48px + полный subtitle). Иконки [≡][≣][▤].

### 4.9. Шкала времени (P0)

- Sticky сверху при вертикальном скролле.
- Адаптивная гранулярность: при зум=1 неделя — дни недели, при зум=1 мес — недели, при зум=3-6 мес — месяцы, при зум=1 год — кварталы.
- **Сегодня** — красная вертикальная линия через всю область, с подписью «сегодня» вверху.
- **Вехи**: маркеры важных дат (например, «открытие сезона на WB» — задаются в админке).
- **Месячные сетки** — светло-серые вертикальные линии.
- **Выходные** — лёгкая заливка в режимах ≤ 3 мес.

### 4.10. Строка заказа (P0)

```
┌─────────────────────────────────────────────────────────────────┐
│ [📷фото] Пальто Классика #1284 · шоколад●  240шт           [⋯]  │
│         Вера · Фабрика «Линь» · White One                       │
│         [Разраб]██[Произв]████████🔥[ОТК]████[Дост]████         │
│                       ↑ tooltip при hover на бар                 │
└─────────────────────────────────────────────────────────────────┘
```

**Левая колонка строки** (sticky при горизонтальном скролле):
- Чекбокс мульти-выбора.
- Фото-превью (с цветовой точкой по `colorName` через `colorHexFromName`).
- Заголовок: фасон · #номер · цвет.
- Subtitle: PM · Фабрика · Бренд · кол-во шт.
- Бейдж статуса (текущая фаза / done).
- Меню `⋯`: открыть, дублировать, скрыть из вида, поставить «не двигать», добавить комментарий.

**Правая часть** (плашки на таймлайне):
- 4 фазы для заказа, 3 для упаковки, 2-4 для разработки фасона (как в текущей версии).
- Цвет — по фазе (Разраб=розовый, Произв=синий, ОТК=янтарный, Доставка=фуксия). Палитру согласуем после первого реализованного скетча.
- **Состояния плашки:**
  - `done` (фаза пройдена) — заливка с opacity 50%, галочка ✓ внутри.
  - `active` (текущая фаза) — full color, лёгкое внутреннее свечение.
  - `future` (ещё не начата) — pattern-заливка диагональными линиями.
  - `overdue` (просрочена) — красный border 2px + иконка 🔥.
  - `nearly-due` (≤ 5 дней до конца) — янтарный border 2px.
- **Drag**: только за края ◀ ▶ (закон, не ломаем). Контракт жестов — см. `HANDOFF.md`.
- **Hover на бар** → tooltip: «Производство · 14 → 28 апр (14 дн) · Фабрика Линь · ответственная Вера». Кнопки в тултипе: «Открыть», «Перенести +N», «Изменить ответственного».
- **Click на бар** → открыть карточку заказа, с автоматическим scroll'ом к секции этой фазы.

**Маркеры рисков справа от строки** (badge area):
- ⚠ просрочка
- 🔥 горит (≤5 дн)
- 🚫 заблокирован (например, упаковка не готова)
- ⏸ на паузе (флаг «не двигать»)

### 4.11. Bulk-операции (P1)

Появляются, когда выбрано ≥ 2 строки (snackbar внизу):
- Перенести все даты ±N дней.
- Сменить ответственного.
- Сменить фабрику.
- Экспорт выбранных.
- Скрыть из вида (временно, на сегодня).

### 4.12. Поведение при пустом состоянии (P0)

- Нет заказов вообще → большая иллюстрация + «Создать первый заказ».
- Фильтры дали 0 → «Ничего не подходит. Сбросить фильтры».
- Все заказы done → «Всё спокойно. На этой неделе ничего не горит 🎉».

### 4.13. Производительность (P0)

- **Виртуализация строк** при > 50 строк (react-window / similar). Без неё на 200 строках лагает на трекпаде.
- **Виртуализация столбцов** не нужна (шкала времени = одна div с absolute-позиционированием).
- Запрос — один `Promise.all` на сервере, как сейчас. Пагинация *внутри страницы* через виртуализацию, не через page=.
- Refresh: кнопка в шапке + `revalidatePath('/gantt')` после каждого PATCH.

### 4.14. Mobile (P1)

На < 768px переключаемся на «Список» автоматически (не Гант). Гант на телефоне не работает — слишком плотно. В списке:
- Карточка как сейчас в `MobileGanttList` (фото, название, дедлайн, мини-bar).
- Sticky-фильтр сверху: только «Горит / Все / Мои».
- Расширенные фильтры — в bottom sheet.
- Свайп влево по карточке → быстрые действия (открыть / +1 нед).

### 4.15. Совместная работа (P2)

- Live-обновление через polling раз в 60с (или WS позже).
- Аватарки команды на барах, если кто-то *сейчас* открыл этот заказ.
- История изменений у каждого заказа («Вера сдвинула ОТК с 12.06 на 18.06»).

## 5. Не-функциональные требования

| Категория | Требование |
|---|---|
| **Скорость** | LCP < 1.5s на 200 заказах. Drag — 60 fps. |
| **Стабильность** | После любого drag — оптимистичный UI, при ошибке — rollback + toast. |
| **A11y** | Все интерактивы доступны с клавиатуры (Tab, стрелки для drag — Shift+→/← двигают на 1 день). |
| **i18n** | Только русский (см. правила проекта). |
| **Правила Алёны** | Без жаргона: «Фаза» а не «pipeline stage», «Дедлайн» а не «deadline» (русифицировать). Цвета в UI — с hex-кружком (memory `feedback_color_chip`). |
| **Фолбэк** | Если drag не сработал (сетевая ошибка) — toast + автоповтор 1 раз. |

## 6. Out of scope (явно НЕ делаем сейчас)

- Печатные расписания / Send to email — есть экспорт.
- Зависимости между заказами (один заказ блокирует другой) — пока нет.
- Ресурсное планирование (загрузка фабрики по человеко-часам) — отдельная вкладка позже.
- WB-интеграция (когда товар на полке) — отдельный модуль.
- Финансовая аналитика — отдельная вкладка `/plan-vs-fact`.

## 7. Контракт жестов в Ганте v2 (согласовано с Алёной 2026-05-10)

**Главный принцип.** Дедлайны фаз — это коммитменты с фабрикой/логистикой. Они **не двигаются сами по себе**. Хвост едет вправо/влево только если предыдущая фаза реально удлинилась/укоротилась (т.е. ▶ потянули).

| Жест | Что меняется в БД | Эффект на UI |
|---|---|---|
| **▶ любой плашки вправо** | endField этой фазы +N | Эта фаза длиннее на N. Все следующие фазы едут вправо на N, их длительности сохраняются. |
| **▶ любой плашки влево** | endField этой фазы -N | Эта фаза короче на N. Хвост едет влево на N, длительности сохраняются. |
| **◀ первой фазы (Разработка)** | startField (decisionDate) ±N | Меняется только старт цепочки. End разработки не двигается → разработка длиннее/короче. **Хвост стоит.** |
| **◀ не-первой фазы** | endField предыдущей фазы (= тот же самый день в БД) | Эквивалент ▶ предыдущей. |

**Дополнительно:**
- **Закрытые (done) фазы редактируются как обычные.** Алёна часто будет заполнять исторические даты задним числом — без подтверждений.
- **Без clamp'ов** — даты можно тянуть и в прошлое, и в будущее на любую величину.
- **Drag-середины-как-move отключён** (закон с прошлой итерации сохраняется).
- **Сдвиг всей цепочки одним жестом не нужен** — такое почти не бывает; если потребуется — двигают по фазам.
- **Plan vs Fact не храним.** В БД лежат актуальные даты; снимков плана нет.

**Что меняется в коде.** Старый закон «◀ первой плашки сдвигает всю цепочку, длительности сохраняются» **отменяется**. Теперь ◀ первой = меняет только start. Это надо переписать в `gantt-chart.tsx` (DraggableBar / onCommitStart) и проверить во всех трёх таймлайнах: `gantt-page-client.tsx`, `order-timeline.tsx`, `packaging-order-timeline.tsx`.

## 10. Acceptance criteria

Считаем v2 готовой, если:

1. Алёна заходит → **за < 30 сек** видит, что горит и принимает решение.
2. На 200 заказах в БД фильтр срабатывает за < 200мс, drag — без лагов.
3. URL шарится: открыл по ссылке → те же фильтры + группировка + зум.

5. Команда переехала на этот экран и не пользуется `/my-tasks` для оперативки.
6. KPI-полоса корректно считает «горит/просрочено/OTD» на тестовых данных.
