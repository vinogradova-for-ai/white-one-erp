## 1. Сквозные принципы UX/UI

### П1. Визуализация вместо текста
Везде, где есть **цвет, товар, упаковка, фото** — видно картинку, а не только слово:
- Цвет — hex-кружочек рядом с названием (`ColorChip`)
- Товар/упаковка — миниатюра фото
- Статус — цветной бейдж, а не текст
- Даты — полосы Ганта, не числа

### П2. Автоматика важнее ручного ввода
Любое поле, которое можно вывести из другого — выводим:
- Платежи → из `paymentTerms` + `totalBatchCost` + `saleStartDate`
- `launchMonth` → из `saleStartDate` (ползунок Ганта)
- `inProductionQty` упаковки → сумма активных `PackagingOrder`
- `stock` упаковки → увеличивается при `PackagingOrder.status = ARRIVED`
- SKU упаковки → автогенерация `BIR-001` / `SZ-001` / `BOX-001`
- Ссылка на фасон в заказе → подтягивает фото, цвет, размеры, себестоимость

**Ручной ввод — только там, где реально есть решение человека.**

### П3. Одна архитектура «заказа» — клонируется на упаковку
Заказ одежды и заказ упаковки работают по одной схеме:
- **Мульти-позиции** (несколько цветов / несколько типов упаковки в одном заказе)
- **Секции формы:** Позиции / Стоимость / Платежи / Параметры / Даты
- **Автоплатежи** на основе суммы и условий оплаты
- **Гант-таймлайн** с этапами
- **Статусы** (ORDERED → IN_PRODUCTION → IN_TRANSIT → ARRIVED)
- **Auto-increment остатка** при ARRIVED

### П4. Форма — одна простыня, без мастеров и прогресс-баров
Если всё автозаполняется и пользователь только корректирует — прогресс-бар бесполезен и визуально мусорит (особенно в тёмной теме). Убираем **везде**.

Форма = одна страница, секции разделены карточками/fieldset. Sticky-кнопка «Сохранить» внизу.

### П5. Даты — только через Гант, не через календарные инпуты
Все ключевые даты (старт производства, доставка, упаковка, ВБ, продажи):
- Тащатся мышкой по временной оси
- Подписи дат всегда видны над полосой (крупные, контрастные)
- При перетаскивании — плавающий ярлык с текущим значением
- Подвижны оба края (и старт, и дедлайн)

Календарные `<input type="date">` остаются только в мелких деталях (дата прихода фактически, дата платежа и т.п.), и только свёрнутые в «Детали».

### П6. Убираем поля, которые не участвуют в процессе
Всё, что не читается никем и ничем не влияет:
- **«Сезон»** (свободный текст, дублирует launchMonth) — убрать
- **«Месяц продаж»** как инпут — убрать (берём из saleStartDate)
- **«Примечание»** в редактировании фасона — убрано
- **«Заметки»** при создании упаковки — убрать
- **«В производстве» (ручной инпут)** в карточке и форме упаковки — убрать (считается автоматом)
- **«Минимум на складе»** для упаковки — убрано
- **ТНВЭД** в форме редактирования фасона — убрано (оставить на заказе, если нужно Элине)
- **Теги у фасона** — уже убраны (миграция `remove_model_tags`)

### П7. Главный экран = один кабинет на всех
`/my-tasks` — один экран, карточки по сотрудникам, все видят всех. Сортировка: кто отстаёт — выше. Никаких персональных/ролевых кабинетов — всё прозрачно. **Не перекидывать** пользователя с главной куда-то насильно.

### П8. Главный Гант = ТОЛЬКО заказы
Образцы, разработка фасонов, упаковка — отдельные процессы с другим ритмом. Смешивать нельзя — визуально непонятно. `/gantt`:
- Только `Order`
- Фильтр по фазам (Производство / Логистика / Упаковка / Поставка ВБ) — одной фазой смотрим, что сейчас в этой стадии
- Миниатюры позиций слева у строки

## 3. Архитектура данных — миграции

### М1. `PackagingOrderLine` (SAME MIGRATION)
```prisma
model PackagingOrderLine {
  id               String  @id @default(cuid())
  packagingOrderId String
  packagingOrder   PackagingOrder @relation(fields: [packagingOrderId], references: [id], onDelete: Cascade)
  packagingItemId  String
  packagingItem    PackagingItem @relation(fields: [packagingItemId], references: [id])
  quantity         Int
  unitPriceRub     Decimal? @db.Decimal(12, 2)
  unitPriceCny     Decimal? @db.Decimal(12, 2)
  priceCurrency    Currency?
  cnyRubRate       Decimal? @db.Decimal(10, 4)
  @@index([packagingOrderId])
  @@index([packagingItemId])
}
```
Backfill: из каждого существующего `PackagingOrder` создать 1 строку и скопировать поля.
После backfill — удалить `packagingItemId/quantity/unitPrice*/cnyRubRate` из `PackagingOrder` (отдельной миграцией или сразу, если безопасно).

### М2. Убрать `inProductionQty` из `PackagingItem`
Это поле руками не должно редактироваться. Считать из SQL: `SELECT SUM(quantity) FROM PackagingOrder WHERE packagingItemId = ? AND status NOT IN ('ARRIVED','CANCELLED')` (после М1 — из `PackagingOrderLine`).

После удаления — везде в коде заменить `item.inProductionQty` на вычисленное значение.

### М3. Auto-increment stock при ARRIVED
В API-рутере `PATCH /api/packaging-orders/[id]` при смене статуса на `ARRIVED`:
```
await prisma.$transaction(async (tx) => {
  for (const line of order.lines) {
    await tx.packagingItem.update({
      where: { id: line.packagingItemId },
      data: { stock: { increment: line.quantity } },
    });
  }
  await tx.packagingOrder.update({ where: { id }, data: { status: 'ARRIVED', arrivedDate: new Date() } });
});
```
Откат при обратной смене статуса — аналогично, с decrement.

## Короткий TL;DR

- **Меньше полей, больше автоматики.**
- **Всё что про заказ — единый паттерн (одежда и упаковка).**
- **Даты — только Гантом.**
- **Главный — все видят всех, никаких редиректов.**
- **Миграции делаем отдельным блоком, не по ходу.**
