# ТЗ: Платёжный календарь

_Составлено 2026-04-17 по итогам разговора с Алёной._

## Зачем

Сегодня команда держит в голове (и в Google-таблицах) кто, когда и за что платит фабрикам и поставщикам упаковки. Нужен единый экран, где видно:
- какие платежи предстоят в ближайшие недели;
- сколько денег нужно в этом месяце;
- что уже оплачено, что просрочено.

## Область

В календаре два типа платежей:

1. **Закуп товара (фабрики).** Предоплаты и постоплаты по заказам на производство. Сумма — из `batchCost` заказа, доли — из условий оплаты («30/70», «100% ПП», «50/30/20» и т.п.).
2. **Упаковка.** Платежи поставщикам упаковочных элементов (бирки, полибэги, чехлы). Сумма и дата заводятся руками, т.к. сейчас в БД цен поставщиков упаковки нет.

**Вне фокуса (пока):** логистика/карго/ВЭД, аренда, зарплаты, реклама, подрядчики. Если понадобится — расширим позже «прочими платежами».

## Откуда берутся платежи

### По заказам на производство (фабрики)

**Гибрид:** система автоматически предлагает график, Алёна/Даша могут подправить любую дату и сумму.

Алгоритм автопредложения:
- На заказе уже есть строковое поле `paymentTerms` («30/70»). Парсим его в массив долей: `[30, 70]`, `[50, 30, 20]`, `[100]`.
- Если строка не парсится — предлагаем одну постоплату 100%, помечаем жёлтым «уточни условия».
- Опорные даты — из заказа:
  - Предоплата привязана к **дате заказа** (или дате согласования с фабрикой, если такое поле появится).
  - Постоплаты — к **дате готовности партии** (`productionReadyAt` или аналог; если нет — к `launchMonth – 45 дней`, уточним на имплементации).
- Суммы считаются от **`batchCost`** (снапшот с момента создания заказа).

Пример: заказ на 2 000 000 ₽, условия 30/70, дата заказа 15.04.2026, готовность 15.07.2026 →
- 15.04.2026 — 600 000 ₽ (предоплата 30%)
- 15.07.2026 — 1 400 000 ₽ (постоплата 70%)

Любой из этих платежей можно вручную подвинуть по дате или пересчитать сумму (например, фабрика согласилась на доплату позже).

### По упаковке

Руками: Настя (или кто ведёт упаковку) в справочнике упаковки или прямо на экране календаря создаёт платёж: «Поставщик ABC — бирки — 180 000 ₽ — 20.04.2026».

Связь с `PackagingItem` опциональная (можно выбрать элемент из справочника, чтобы потом видеть историю закупов, но без привязки тоже ок).

## Что должен уметь экран

**Путь:** `/payments` в левом меню («Платежи»).

Два представления через переключатель:

### 1. Список (лента)

Таблица/карточки платежей в хронологическом порядке. Колонки:
- Дата планируемая
- Тип (фабрика / упаковка)
- Контрагент (фабрика или название поставщика)
- За что (номер заказа + фасон + цвет, или название упаковки)
- Сумма
- Статус (ждёт / оплачено)
- Действия (отметить оплаченным, редактировать, удалить)

**Фильтры:** месяц, тип (фабрика/упаковка), статус, конкретная фабрика/поставщик.

**Группировка по умолчанию:** по неделе (эта неделя / следующая / позже).

### 2. Календарь-сетка

Классический месячный календарь. В клетке дня — чипы платежей (контрагент + сумма). Клик по чипу — попап с деталями и кнопкой «отметить оплаченным».

Переключение месяцев стрелками. Сверху итог месяца: «К оплате в апреле: 4 300 000 ₽, из них оплачено 1 200 000 ₽».

### Общее для обоих видов

- **Сумма в шапке экрана:** «К оплате до конца месяца: X ₽» — крупно, чтобы Алёна видела сразу.
- **Просрочка:** если дата платежа < сегодня, статус = ждёт → красный бэйдж «просрочено на N дней».
- Блок «Дефицит денег на неделю» — сумма всех непогашенных платежей, у которых дата ≤ +7 дней.

## Статусы платежа

Минимально:
- `PENDING` — ждёт оплаты
- `PAID` — оплачен (сохраняется `paidAt` = дата фактической оплаты)

На карточке платежа можно:
- изменить плановую дату, сумму, контрагента, комментарий;
- прикрепить ссылку на счёт (одно поле — URL);
- оставить заметку;
- отметить оплаченным (выставляется `paidAt` = сегодня, можно подправить).

Если позже понадобится «в работе» (отправлено в банк) — добавим третий статус. Пока не усложняем.

## Модели в БД

### Новая таблица `Payment`

```
Payment
├── id
├── type           (ORDER | PACKAGING)
├── status         (PENDING | PAID)
├── plannedDate    DateTime    // когда должны заплатить
├── paidAt         DateTime?   // когда реально заплатили
├── amount         Decimal     // сумма в рублях
├── currency       (RUB | CNY)  — позже, пока только RUB
├── label          String      // «Предоплата 30%», «Постоплата», «Бирки — апрель»
├── invoiceUrl     String?     // ссылка на счёт
├── notes          String?
│
├── orderId        String?     // если type = ORDER
├── factoryId      String?     // дубль для быстрых фильтров
│
├── packagingItemId String?    // если type = PACKAGING
├── supplierName   String?     // контрагент-поставщик упаковки (свободный текст)
│
├── createdById
├── createdAt
├── updatedAt
└── paidById       String?
```

### Изменения существующих моделей

- `Order` — появится связь `payments: Payment[]`. Больше ничего. Текстовое `paymentTerms` оставляем как есть (оно — источник для парсера).
- `PackagingItem` — появится связь `payments: Payment[]`.
- `Factory` — появится связь `payments: Payment[]` (через orderId косвенно, но удобно для фильтра).

### Автосоздание платежей

На API-ручке создания заказа (`POST /api/orders`) после создания заказа парсим `paymentTerms`, создаём нужное число `Payment` с `status = PENDING` и рассчитанными суммами/датами. Если парсер не справился — создаём один платёж на 100% с пометкой «уточните условия оплаты» в notes.

При изменении `batchCost` или `paymentTerms` на заказе — **НЕ пересоздаём** автоматически (чтобы не затереть ручные правки). Вместо этого показываем на карточке заказа предупреждение «Сумма изменилась, платежи не пересчитаны» + кнопка «Пересчитать платежи» (удаляет `PENDING`-платежи и создаёт заново, `PAID` не трогает).

## Права (RBAC)

| Действие | OWNER | DIRECTOR | PM | ASSISTANT | LOGISTICS | CUSTOMS |
|---|---|---|---|---|---|---|
| Видеть календарь | ✓ | ✓ | ✓ (свои заказы) | ✓ (упаковка) | ✓ (своё, позже) | ✓ (своё, позже) |
| Создать платёж вручную | ✓ | ✓ | ✓ | ✓ (только упаковка) | — | — |
| Редактировать дату/сумму | ✓ | ✓ | ✓ (свои) | ✓ (упаковка) | — | — |
| Отметить оплаченным | ✓ | ✓ | — | — | — | — |
| Удалить | ✓ | ✓ | — | — | — | — |

Детали по ассистентам уточним, когда дойдём до реализации.

## Этапы реализации

1. **БД и миграция.** Новая таблица `Payment`, связи. Сид-данные для демо.
2. **Парсер `paymentTerms`.** Функция `parsePaymentTerms("30/70")` → `[0.3, 0.7]`. Тесты.
3. **Автосоздание при создании заказа.** Дополнить `POST /api/orders`. Кнопка «Пересчитать платежи» на карточке заказа.
4. **Ручной CRUD платежей.** API + форма создания/редактирования для упаковки и для произвольного платежа по заказу.
5. **Экран `/payments` — список.** Фильтры, группировка, действия. Это ядро.
6. **Экран `/payments` — календарь-сетка.** Переключатель. Клик по чипу → попап.
7. **Сводки.** Цифры в шапке, блок «дефицит на неделю», пункт в дашборде «Что горит».
8. **«Мои задачи».** Для Алёны/Даши — «Сегодня платить N на сумму X». Для Насти/Тани — свой срез.

Каждый этап — отдельный коммит, показываю результат, двигаемся дальше.

## Что точно НЕ делаем в этой итерации

- Интеграцию с банком/1С/Excel.
- Мультивалютность (CNY-платежи фабрике в юанях). Курс фиксируем на создании, суммы в рублях.
- Повторяющиеся платежи (аренда, подписки).
- Уведомления в Telegram о сроках оплаты — это фаза 4 (бот).
- Отчёты «факт vs план по месяцу» — отдельная задача.
- Платежи по логистике и ВЭД — добавим позже, когда устаканим базовый флоу.

## Открытые вопросы

1. **Опорная дата постоплаты.** Есть ли на заказе дата готовности партии (`productionReadyAt` или аналог)? Если нет — использовать `launchMonth – 45 дней`? Проверю в схеме на шаге 1 и приду с точным ответом.
2. **Упаковка оптом vs поштучно.** Платим за партию бирок или по конкретным заказам? Скорее за партию — тогда `packagingItemId` связь опциональная, платёж живёт сам по себе.
3. **Статус просрочки.** Автоматом красный при `plannedDate < сегодня` + `status = PENDING`, либо отдельный статус `OVERDUE`? Предлагаю автоматом, без отдельного статуса — меньше сущностей.
4. **Удаление PAID-платежей.** Разрешить только OWNER/DIRECTOR с комментарием, по аналогии с откатом статусов заказа. Согласовать отдельно.
