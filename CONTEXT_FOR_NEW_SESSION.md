## 🏗 Архитектура v2 (актуальная — экономика на фасоне)

### Ядра

- **ProductModel (Фасон)** — «Пальто Классика Миди». Один фасон = N цветовых вариантов. На фасоне живут:
  - Название, бренд, категория, теги, страна, фабрика по умолчанию
  - Ткань (название/состав/расход/цена) — опционально
  - Размерная сетка (справочник)
  - ТНВЭД (таможенный код, один на все цвета)
  - **Экономика**: purchasePrice (CNY/RUB), packagingCost, wbLogisticsCost, wbPrice, customerPrice, commission, DRR, plannedRedemption. Одна на все цвета.
  - Расчёты: fullCost, margin, ROI, markupPct
  - `patternsUrl` — **одна** ссылка на папку с материалами (лекала, тех-пакет, фото образцов — всё вместе)
  - `photoUrls` — общие фото фасона
  - `targetCostRub` / `targetCostCny` — прогноз до согласования с фабрикой
  - `turnkeyPurchase` — флаг «закупка под ключ»
  - **Комплект упаковки** (ModelPackaging) — шаблонный список, который автоматически копируется в OrderPackaging при создании заказа

- **ProductVariant (Цветовой вариант)** — «шоколад», «беж», «чёрный». На варианте:
  - SKU (уникальный, при soft-delete префиксуется чтобы освободить)
  - Цвет, `fabricColorCode` (артикул цвета у поставщика ткани, для повторного заказа)
  - Фото (≥1 обязательно)
  - `factRedemptionPct` — **единственное экономическое поле** (реальный % выкупа из WB, отличается по цветам)
  - Габариты упаковки (lengthCm, widthCm, heightCm, weightG, liters) — при редактировании
  - `packagingType` — старое поле строкой (название типа), не M2M

- **Order (Заказ на производство)** — партия на конкретный variant. Все даты, QC, упаковка (через OrderPackaging), ВЭД, оплаты, снимки экономики с фасона в момент создания.

- **Sample (Образец)** — канбан REQUESTED → IN_SEWING → DELIVERED → APPROVED → READY_FOR_SHOOT → RETURNED.

- **Idea** — пул идей с приоритетом, тегами, вдохновляющими фото (`imageUrls`). Промо → создаёт ProductModel.

- **Factory** — справочник (название, страна, город, контакт, мощность).
- **SizeGrid** — справочник размерных сеток (свободный ввод, парсер диапазонов: `40-60`, `XS-XXL`, `40-46 step 1`).
- **PackagingItem** — справочник упаковки (бирки, размерники, полибэги, чехлы). Остатки + «в производстве» + минимум.
- **ModelPackaging** — M2M модель↔упаковка (шаблон для заказов).
- **OrderPackaging** — M2M заказ↔упаковка (фактическая потребность).
- **User, MonthlyPlan, AuditLog** — прочие.

### Статус-машины

- **ProductModel**: IDEA → PATTERNS → SAMPLE → APPROVED → IN_PRODUCTION
  - **Жёсткие проверки** (новое в этой итерации):
    - → SAMPLE требует ≥1 фото
    - → APPROVED требует: экономика заполнена (fullCost есть), размерная сетка, фото
    - → IN_PRODUCTION требует: хотя бы 1 вариант в READY_TO_ORDER
  - Откат только OWNER/DIRECTOR с комментарием
- **ProductVariant**: DRAFT → READY_TO_ORDER / DISCONTINUED
- **Order**: 10 статусов от PREPARATION до ON_SALE
  - **Жёсткая проверка**: → PACKING требует привязанной упаковки без дефицита (total ≤ stock + inProduction)
- **Sample**: 6 статусов drag-and-drop на канбане. APPROVED требует комментарий.

### Ключевые бизнес-решения (не ломать)

1. **Бренды** — enum `Brand` (WHITE_ONE / SERDCEBIENIE), не тег.
2. **Теги на фасоне** — массив строк с автоподсказкой.
3. **«Закупка под ключ»** — флаг `turnkeyPurchase`: если true, поля ткани игнорируются.
4. **Таргет себестоимости** — прогноз на этапе разработки; актуальная себестоимость — `fullCost`, рассчитывается из заполненных полей закупки/упаковки/логистики.
5. **Валюта закупки** — CNY для Китая (с буфером 5% через `COST_BUFFER_PCT`) + курс, RUB для России.
6. **Фабрика на фасоне** — «по умолчанию», в заказе можно поменять.
7. **Часовой пояс** — БД UTC, UI MSK.
8. **Фото** — пока только URL-ссылки + локальный drag-n-drop в идеях (папка `public/uploads/`). Drag-n-drop для фасонов/цветов — через ссылки или облако в будущем.
9. **Размерная пропорция** — НЕ на фасоне, НЕ на варианте. При создании заказа штуки раскладываются автоматически, вручную можно переопределить. Поле `defaultSizeProportion` в БД оставлено как неиспользуемое.
10. **SKU варианта** — уникальный. При soft-delete префиксуется `__deleted_{timestamp}`, чтобы SKU можно было переиспользовать.
11. **ТНВЭД** — на фасоне (один для всех цветов).
12. **Артикул цвета у поставщика ткани** — на варианте (`fabricColorCode`), заменил Pantone.

## ✅ Что работает

- CRUD фасонов / вариантов / заказов / образцов / идей / упаковки / фабрик
- Админки: `/admin/users` (просмотр), `/admin/factories` (CRUD с архивом)
- Дашборд «Что горит сейчас» — просроченные заказы, застрявшие образцы/фасоны, дефицит упаковки, разрыв плана
- Форма создания фасона минимальная: основное + производство + ткань + фото + документы + **комплект упаковки** + примечания. Экономика и размерная пропорция — НЕ здесь.
- Форма редактирования фасона: всё то же + экономика (заполняется после согласования с фабрикой)
- Форма создания цвета минимальная: артикул + цвет + артикул цвета у поставщика ткани + фото + примечания. Пропорция/габариты/факт-выкуп — на редактировании.
- Drag-and-drop фото в идеях (локально в `public/uploads/`)
- Размерные сетки: свободный ввод, парсер диапазонов
- Стрелки ↑↓←→ в таблице распределения размеров в заказе
- «Мои задачи» по ролям + задачи по упаковке (Настя/Таня/админы видят «запустите производство — не хватает N шт для заказов X, Y»)
- Жёсткие проверки при смене статуса (см. выше)
- Автокопирование комплекта упаковки с фасона в новый заказ
- Окна смежных отделов: `/content-schedule` (Катя), `/incoming` (Таня), `/customs` (Элина), `/warehouse-receipt` (Настя)
- План/Факт по месяцам · Загрузка фабрик (+ кнопка в справочник)
- Аутентификация: после пересоздания БД битый токен автоматически редиректит на /login

## 🐛 Найденные и починенные баги (за последний заход)

Алёна ввела шуточный штраф 1 000 ₽ за каждый найденный баг.

1. **Zod `.default()` в partial-схемах обнулял поля при PATCH.** Самый коварный: правка экономики на фасоне тихо снимала галочки «повтор», «закупка под ключ», «корректировки», сбрасывала теги и фото. Фикс: убрано `.default()` из create-схем (model, idea, packaging), дефолты теперь задаются в формах и @default в Prisma.
2. **Soft-delete варианта держал SKU занятым.** Команда удаляет цвет — не может создать новый с тем же артикулом. Фикс: при DELETE SKU префиксуется `__deleted_{timestamp}`.
3. **Login page ломал прод-сборку.** `useSearchParams()` нужен `<Suspense>`. Фикс: обёрнуто.
4. **Битый токен после reset БД → ошибка «Доступ запрещён».** Layout пропускал пользователя с сессией, где role=undefined. Фикс: проверка наличия id/role в session.user, redirect на /login.

Долг на 2026-04-17: **4 000 ₽**. Алипей.

## 🔒 Правила

1. **НЕ пересоздавать БД** без явного разрешения (использовать `PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION` с точной цитатой). Алёна выдала консент: «да там все данные фейковые забиты, мы не боимся их потерять».
2. **НЕ `git push --force`** на main.
3. **НЕ добавлять** функциональность вне фокуса продуктового отдела (WB-метрики, HR, мудборды, факт-маржа по цвету).
4. **Коммиты** — осмысленные.
5. **Русский язык** кроме кода.
6. **Никаких `.default()`** в Zod-схемах, которые используются с `.partial()` для PATCH.
7. **Ссылки на файлы** — markdown-ссылки `[path](path:line)`, Алёна кликает.
