## 📦 Что сделано в этой сессии (кратко)

### 1. Платёжный календарь (ТЗ в `docs/payment-calendar-spec.md`)
- Новая таблица **Payment** (миграция `add_payment_calendar`). Связи с Order, Factory, PackagingItem.
- Парсер `paymentTerms` (`src/lib/payments/parse-terms.ts`) — «30/70» / «50/30/20» / «100».
- Автосоздание платежей при создании заказа (в `POST /api/orders`).
- Экран **/payments** — список с фильтрами (месяц / тип / статус), сводки «К оплате / Оплачено / Просрочено», кнопки «Оплачен/Откатить/Править/Удалить».
- Формы `/payments/new` и `/payments/[id]/edit` — ручное создание/правка, вкладки Фабрика/Упаковка.
- API: `POST/GET /api/payments`, `PATCH/DELETE /api/payments/[id]`, `POST/DELETE /api/payments/[id]/mark-paid`.
- Дашборд: блок «Платежи этого месяца» + секция «Просроченные платежи» в «Что горит».
- Вырезано сознательно (не делать без запроса): календарь-сетка, «Мои задачи» по платежам, поле `invoiceUrl` в UI, поле `paidBy` в UI, мультивалютность, связь с PackagingItem в UI.

### 2. Мультипозиции в заказе (один фасон — несколько цветов)
- Новая таблица **OrderLine** (миграция `multi_line_orders`). В Order переехал `productModelId` (фасон общий), все экономические поля и количество — на OrderLine.
- БД пересоздана (`migrate reset`), seed обновлён — теперь у каждого заказа несколько позиций.
- Форма `/orders/new` переделана: выбор фасона → табличка цветов с количеством и размерной матрицей у каждой позиции.
- Страница `/orders/[id]` показывает позиции, в заголовке — фото фасона + общая инфо. Редактирование позиций (кол-во, размеры, удаление, добавление новой) — через компонент `OrderLinesSection`.
- Форма `/orders/[id]/edit` — только шапка заказа, позиции правятся на странице заказа.
- API переделан: `POST /api/orders` принимает `productModelId + lines[]`; `POST /api/orders/[id]/lines`, `PATCH/DELETE /api/orders/[id]/lines/[lineId]`.
- Список заказов и дашборд адаптированы.

### 3. Форма создания заказа — дополнительно
- Поле **«Стоимость единицы, ₽»** (по умолчанию = `model.fullCost`, можно переопределить) — передаётся как `unitCost` на бэк и переопределяет экономику линий.
- Блок **«График платежей»** — редактируемые строки (дата/название/сумма), автогенерация по `paymentTerms`, подсвечивается расхождение с суммой заказа.
- Блок **«Упаковка на партию»** — подтягивается с фасона, показывает потребность и дефицит.
- Блок **«Таймлайн изделия»** (`OrderTimeline`) — 7 этапов с цветными точками: Производство 🪡 / ОТК ✓ / Доставка ✈ / Съёмка 📷 (ориентир) / Упаковка 📦 / Поставка на ВБ ⬆ / Старт продаж 🚀. Даты авто-раскладываются от сегодня до 1-го числа launchMonth, редактируемые.
- **Убрано:** поле «Условия оплаты» (дубль графика), «Примечания» (не нужно).
- Month-picker вместо ввода YYYYMM.

### 4. Упаковка — полный жизненный цикл (миграция `packaging_lifecycle`)
- Enum `PackagingItemStatus`: IDEA → DESIGN → SAMPLE → APPROVED → ACTIVE → ARCHIVED.
- Поля стоимости: `unitPriceRub`, `unitPriceCny`, `priceCurrency`, `cnyRubRate`.
- Даты этапов + `ownerId` + таблица `PackagingItemStatusLog`.
- Машина переходов `src/lib/status-machine/packaging-statuses.ts` + API `PATCH /api/packaging/[id]/status` с откатом от OWNER/DIRECTOR.
- На карточке упаковки: статус-бэйдж, кнопка «Сменить статус», блок «Этапы разработки» (5 дат), блок «Стоимость единицы» (с конверсией ¥→₽), история статусов.
- На карточке фасона в «Комплекте упаковки» — стоимость на единицу товара + итого.
- Автосоздание `Payment` (тип PACKAGING) при переходе в ACTIVE — если есть цена и `inProductionQty > 0`.

### 5. Визуал цветов (без миграции)
- Словарь «цвет → hex» в `src/lib/color-map.ts` (~60 популярных цветов).
- Компонент `VariantVisual` (`src/components/common/variant-visual.tsx`): если у варианта есть своё фото → показываем; если нет, но есть фото фасона → фото фасона + цветной кружок в углу; если нет ничего → большой квадрат цвета.
- Заменено в списках вариантов, заказов, /incoming, /warehouse-receipt, /content-schedule, на странице заказа.

### 6. Ночной режим

## ⚙️ Правила (из `.claude/CLAUDE.md` и по ходу сессии)

1. **Никаких `.default()`** в Zod-схемах, используемых с `.partial()` для PATCH — обнуляет поля.
2. **Не пересоздавать БД** без явного разрешения пользователя. Алёна выдавала консент: «да там все данные фейковые забиты, мы не боимся их потерять».
3. **Простой язык**, никакой технической простыни. Говорить «форма / справочник / карточка / ссылка», а не «API / JWT / миграция».
4. **Короткие отчёты** после правок: «Готово. Обнови страницу. Теперь работает X.»
5. **Коммиты** — не делать без запроса.
6. **`git push --force`** на main — нельзя.

## 🐛 Найденные и починенные баги в эту сессию

1. **Dev-сервер держал старый Prisma Client** после миграций. Решение — рестарт `npm run dev` (я делала это дважды через `kill + nohup`).
2. **Tailwind v4 и dark mode:** кастомные CSS-правила проигрывали утилитам по специфичности — пришлось использовать `!important`.
3. **Палитра iOS dark mode** — сначала сделал слишком насыщенные tint'ы, получились «коричневые карточки». Убавил до 5–7% alpha.
4. **Primary-кнопка `bg-slate-900`** сливалась с фоном в dark — переопределил на `#5a5a5e` для button/a.
5. **`.next/types/validator.ts` держал старые импорты** удалённых страниц (`/ideas/*`) — лечится очисткой `rm -rf .next`.
